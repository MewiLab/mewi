# Technical Design Note — Perception-Memory Pipeline

**Component:** `mewi/backend`
**Last updated:** 2026-04-19
**Scope:** Unity snapshot → LLM prompt → Action → Async memory log

---

## 1. Perception Pipeline

Unity sends a JSON snapshot on every tick via `POST /api/v1/agent/tick`. The data flows through three stages before it reaches the LLM:

```
Unity JSON
    │
    ▼
SnapshotManager.process()          app/agent/perception.py
    • Validates with Pydantic (CreatureSnapshot + EnvironmentSnapshot)
    • Filters entities beyond relevance_radius (default 30 m)
    • Assesses ThreatLevel: SAFE / CAUTION / DANGER
    • Returns PerceptionSummary (or PerceptionError on bad data)
    │
    ▼
PerceptionSummary.to_prompt_context()   app/agent/schemas/perception_schema.py
    • Serialises to a flat dict: tick, threat_level, creature, environment,
      nearby_entities, entity_count
    │
    ▼
format_perception_for_prompt(ctx)       app/agent/prompts.py
    • Renders the dict as structured English prose
    • Falls back to "The environment is calm…" when ctx is empty
    │
    ▼
LLM system + user message              app/agent/graph.py → make_reason_node
```

`AgentService.snapshot_to_prompt(raw)` exposes this same path as a static utility — useful for evals and prompt-cache warm-up without side effects (no tick counter increment, no memory write).

## 2. Instinct-Driven Brain

### Why we moved away from Survival Rules

The previous system prompt used an explicit ordered rule list:

```
1. Threat DANGER  → flee immediately
2. Threat CAUTION → increase distance
3. Threat SAFE    → explore
```

This works for a finite rulebook but breaks down as behaviour complexity grows:
- Rules create hard if/then branches the LLM applies literally rather than weighing context.
- They express **what to do**, not **why**, so the LLM has no basis for creative adaptation.
- Adding new behaviours requires editing the prompt, not just the agent's internal state.

### The instinct model

`MEW_SYSTEM_TEMPLATE` in `app/agent/prompts.py` replaces rules with *instincts*:

| Instinct | Trigger | Typical output |
|---|---|---|
| Self-Preservation | Threat within 5 m | Flee or fight |
| Curiosity | Novel entities / unexplored space | Investigate, annotate |
| Social Bonding | Familiar, friendly entities nearby | Approach, idle close |
| Rest & Recovery | Low energy after long activity | Find shelter, wait |

Instincts are evaluated in priority order but the LLM weighs the full context — a cornered cat with high arousal may override self-preservation with aggression. The prompt also injects the agent's live `goal`, `energy`, `hunger`, and `mood` so decisions reflect transient internal state, not just the external scene.

`build_mew_system_prompt(action_desc, goal, internal_state)` in `prompts.py` renders the template at runtime. All prompt content lives in `prompts.py`; `graph.py` contains only wiring.

## 3. Memory Logging

### Stable identity with UUID v5

After each LLM decision the result is written to the `micrologs` Supabase table. That table requires a `user_id (UUID)`. Because these entries are generated by the agent — not a human user — we derive a deterministic UUID:

```python
_CREATURE_NS = uuid.UUID("a1b2c3d4-e5f6-7890-abcd-ef1234567890")
system_user_id = str(uuid.uuid5(_CREATURE_NS, creature_id))
```

UUID v5 is a SHA-1 hash of `(namespace, name)`. For the same `creature_id` it always produces the same UUID, which means:
- No extra DB row is needed just to hold the system identity.
- Logs for the same creature are always grouped under the same stable ID.
- The namespace UUID acts as a domain separator so creature IDs never collide with real user UUIDs.

### Contextual Retrieval framing

Each log record prepends a situational context header to the decision text — this is Anthropic's **Contextual Retrieval** pattern: chunks stored for later retrieval are self-contained because context is embedded in the chunk itself, not only in surrounding records.

```
[Context: tick=42 | threat=DANGER | pos=(5.0, 8.0) | weather=cloudy | entities_nearby=2]

Action: move
Reasoning: Wolf at 2.8 m — fleeing diagonally to avoid predator.
```

When this record is retrieved out of order (e.g. by a semantic search over the agent's history), the header gives enough context to interpret the decision without needing adjacent records.

### Async fire-and-forget

The log call must never block the Unity polling loop. In `agent_worker.run_agent_job`:

```python
# After svc.complete_job() — Unity can now poll its result
if supabase is not None:
    asyncio.create_task(
        log_contextual_decision(
            action=..., reasoning=..., perception_ctx=..., supabase=supabase
        )
    )
```

`asyncio.create_task` schedules the coroutine on the running event loop without `await`-ing it. The job result is already in Redis before the log write even starts. `log_contextual_decision` swallows all exceptions internally, so a Supabase outage has zero impact on gameplay.

## 4. Running the Tests

The test file requires no external services — LLM, Redis, and Supabase are all mocked.

```bash
# From mewi/backend/
uv run pytest tests/integration/test_perception_flow.py -v
```

### Test classes

| Class | What it covers |
|---|---|
| `TestSnapshotToPrompt` | `AgentService.snapshot_to_prompt()` — text output, threat detection, error handling |
| `TestPerceptionToActionPipeline` | Full LangGraph tick with mock LLM — valid action, threat capture, markdown JSON fence recovery, `goal`/`internal_state` passthrough, bad-payload safety |
| `TestContextualMemoryLog` | `log_contextual_decision()` — DB write, content format, error isolation, valence/arousal bounds |
| `TestRunAgentJob` | End-to-end worker — success path, graph error fallback, `supabase=None` graceful skip |

### Running a single class

```bash
uv run pytest tests/integration/test_perception_flow.py::TestPerceptionToActionPipeline -v
```

### Running alongside the full unit suite

```bash
uv run pytest tests/unit/ tests/integration/test_perception_flow.py -q
```

Integration tests that hit real external APIs are guarded by `@pytest.mark.paid` and require `CONFIRM_PAID=1` — the perception flow test does **not** use that marker.
